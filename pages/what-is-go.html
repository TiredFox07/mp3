What is Go?

Go is a general purpose, garbage-collected, and strongly typed programming language with built-in models for building concurrent programs with specifically systems programming in mind [2]. Go was designed because of frustrations had with the current technologies of the time [3].

Go has an emphasis on simple syntax [2]. Someone with minimal knowledge of the language should be able to parse most of the code written in the language.


		
The given example is simple, though it shows how to declare a package, import external functions, create and assign variables, and print text to the console. Despite not knowing how to write any Go code, you can make assumptions that give you an understanding of what code the code is doing. Go was designed with this principle in mind.


		
Go also has a unique approach to handling errors.

		

		
In this example, a divide function is called. However, instead of crashing the program when the denominator is zero, it simply returns an error. By convention, no error is represented with nil. So after every function call that has a chance to produce an error, we check if one actually occurred. If so, we handle it however we want, and if not, we resume the rest of the program as usual. Many developers have expressed that they feel that Go's error handling is too verbose [4]. However, Go's error handling was designed this way to be as explicit as possible. Without this explicitness, some functions could produce errors that are never caught and thus crash the program.


		
Another defining feature of Go is how it handles concurrency within programs.

		

		
In this example, we call two functions concurrently with the go keyword which creates a goroutine. Note that concurrency is not parallelism. While all code that is running in parallel is concurrent, not all concurrent code is being ran in parallel. Go's scheduler will decide if a goroutine is ran in parallel or not depending on a variety of factors. If a goroutine is not ran in parallel, the Go scheduler will jump between different tasks in order to complete them all. In this case, however, only one part of the code is being executed at once, and thus the code is not running in parallel. If Go's scheduler wants to run two goroutines in parallel, then it will run each goroutine on a processor thread each.
